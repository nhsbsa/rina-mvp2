Content - Content principles

In order to make EESSI RINA structured electronic documents (SEDs) more accessible and bring them closer in style to NHS forms we recommend using the following 7 rules: 

1. Break SEDs into sections

There is no recommended page length in government or NHS style guidance. Public-facing GOV.UK forms are often broken down into single-question screens. 

This does not feel appropriate for our expert users. However, some SEDs are very long and we recommended breaking them down to 1 section per page in order to:

  • improve usability and accessibility
  • bring them closer to NHS style

The SEDs contain between 1 and 7 numbered sections.

We recommend putting each numbered section onto its own page in order to:

  • make them easier to read and navigate
  • maintain a consistent, repeatable approach

Navigation between sections

As the SEDs are broken into sections, users need a way to navigate to the next page.

We recommend using the primary button from the NHS design system with the wording ‘Save and continue’.

Please note: We will look to review this as part of our work to understand how draft functionality works in OHS RINA.

H001 and H002

These are the longest SEDs and only have 1 main section each. They are broken up according to this logic: 1.1.1, 1.1.2, 1.1.3 and so on up to 1.1.8.

We recommend breaking down these 2 SEDs using the third number in the numbering structure. This will mean that both H001 and H002 will be broken into 8 pages.

Source:

Writing for GOV.UK

2. Remove all numbering

Numbering in forms is not recommended in the NHS style guide. We recommend removing all numbering from SEDs, including numbers in ‘Select’ components.

While the numbers help to structure the data, they offer a poor user experience in terms of accessibility and readability.

Source:

NHS design system - How to write good questions for forms

3. Mark mandatory sections and fields with an asterisk

The NHS design system recommends labelling optional fields rather than mandatory ones: 'We do not use asterisks for essential fields. Research suggests that they're not necessary and they can make users anxious and less likely to complete the form.'

However, our users are expert users who interact with SED forms many times each day. There is no evidence that using forms with mandatory fields is a problem for them.

H001 and H002 forms also contain many more optional fields than mandatory ones. Marking mandatory sections and fields (as RINA currently does) rather than optional ones, makes more sense for these forms and allows a consistent approach across these and other SEDs.

Source:

NHS design system - Make sure you need each question

4. Use improved error messages for each component

In the NHS and GOV.UK design systems, error messages are specific to the content of each component.

In the existing SEDs, there is a single error message for all components: ‘requires data’.

As a practical compromise we suggest individual error messages for each type of component in the SEDs.

This is for required fields only. As we build in automation we may need to explore more data validation.

Text input: Enter {field name}

Radios: Select 1 option for {field name}

Checkboxes: Select 1 or more options for {field name}

Date input (where no date has been provided): Enter {field name}

Date input (where 1 field is missing): {field name} must include a day/month/year

Textarea: Enter {field name}

Autocomplete dropdown: Select 1 option for {field name}

Sources:

  • NHS design system - error message
  • GOV.UK design system - error message

5. Formatting

To support accessibility, every page should only have one main heading (h1).

Each main section of the page (these are numbered 1. 2. 3. and so on) should have an h2. Sub-sections have h3 headings then continue down to h6, following the structure of the data.

We recommend adding this override class to h5 and h6 headings:

<h5 class="nhsuk-heading-s"></h5>

Page breaks

We recommend using:

  • an extra large (xl) page break before each h2 section
  • a medium (m) page break before each h3 section

No other section breaks should be displayed.

Sources:

  • NHS design system - typography
  • NHS design system - use headings correctly

6. Use sentence case for headings, sub-headings and field labels

Currently headings and sub-headings in the SEDs are in capital letters. This makes them difficult to read and is not recommended in the NHS design system.

Some field labels also have capital letters at the start of each word, for example, 'Start Date'.

We recommend that all headings, sub-headings and field labels in SEDs are written in sentence case (upper and lower case) with only the first word taking a capital letter.

So ‘PERSON IDENTIFICATION’ becomes ‘Person identification’ and 'Start Date' becomes 'Start date'.

Sources:

  • NHS design system - capitalisation

7. Remove hint text

Currently hint text in the SEDs is hidden in blue information icons. It is revealed when you hover over the buttons with your cursor. This pattern would fail accessibility testing.

As best practice the NHS design system recommends placing hint text on the components themselves. Hint text should be as short and concise as possible.

User research carried out by the RINA UCD team shows that our expert users are able to navigate OHS screens and SEDs with no hint text.

User research carried out by the member state claims team confirms that our users need minimal hint text.

However, if we need to retain the hint text for training or reference purposes, it can be moved to supporting documentation.

Source:

  • NHS design system - hint text
